# Railway Deployment

## Architecture

Railway project with 8 services, all deploying from the same GitHub repo:

| Service | Type | Root Dir | Public |
|---------|------|----------|--------|
| `scoresheet-backend` | Web | `backend` | No (internal only) |
| `scoresheet-frontend` | Web | `frontend` | Yes (public domain) |
| PostgreSQL | Database | — | No |
| `daily-mlb-ingest` | Cron | `backend` | No |
| `news-scrape` | Cron | `backend` | No |
| `il-status` | Cron | `backend` | No |
| `weekly-roster-sync` | Cron | `backend` | No |
| `draft-monitor` | Cron | `backend` | No |

**Internal networking:** Services communicate via `*.railway.internal` hostnames. The backend has no public domain — only the frontend is exposed. Security: API key middleware + CORS + no public URL.

## Railpack Gotchas

Railpack is Railway's builder (replaces Nixpacks). Key things learned the hard way:

### Python packages require `requirements.txt`

Railpack does NOT recognize `pyproject.toml` (setuptools) for dependency installation. Without `requirements.txt`, **zero packages are installed** — no alembic, no uvicorn, nothing. The `pyproject.toml` is still used for `pip install -e .` locally and for pytest config.

**Rule:** Always keep `backend/requirements.txt` in sync with `pyproject.toml` dependencies. When adding a new dependency, update both files.

### Python runtime uses mise

Railpack installs Python via [mise](https://mise.jdx.dev/) (a polyglot version manager). The Python binary lives at `/mise/installs/python/<version>/bin/python`. The PATH includes `/mise/shims` which provides `python` but NOT pip-installed CLI tools.

Once `requirements.txt` exists, Railpack properly installs packages and puts CLI scripts (uvicorn, alembic) on PATH. Without it, the shims only provide the bare Python interpreter.

### Pre-deploy commands

The `preDeployCommand` in `railway.toml` runs after build but before the service starts. It runs in the same container image as the deployed service. The `railway.toml` takes precedence over Railway Dashboard settings — if both are set, the toml wins.

## Config Files

### `backend/railway.toml`

```toml
[build]
builder = "RAILPACK"
watchPatterns = ["backend/**"]

[deploy]
preDeployCommand = "alembic upgrade head"
healthcheckPath = "/api/health"
healthcheckTimeout = 300
restartPolicyType = "ON_FAILURE"
```

- No `startCommand` here — set per-service in Railway Dashboard to avoid cron services inheriting it
- `watchPatterns` prevents rebuilds when only frontend files change
- `preDeployCommand` runs Alembic migrations before every deploy (no-op if none pending)
- Health check ensures zero-downtime deploys: new instance must pass before traffic swaps

### `frontend/railway.toml`

```toml
[build]
builder = "RAILPACK"
watchPatterns = ["frontend/**"]

[deploy]
startCommand = "npx next start -H 0.0.0.0 -p ${PORT:-3000}"
healthcheckPath = "/login"
healthcheckTimeout = 300
restartPolicyType = "ON_FAILURE"
```

- `next start` does NOT read the `PORT` env var automatically — must pass `-p ${PORT:-3000}` explicitly
- `-H 0.0.0.0` binds to all interfaces (required for containers)
- Do NOT use `output: "standalone"` in `next.config.mjs` — it doesn't serve static assets without extra config

### `backend/requirements.txt`

Must mirror `pyproject.toml` `[project.dependencies]`. This is what Railpack reads to install packages. If they drift apart, Railway will have different packages than local dev.

## DATABASE_URL Normalization

Railway provides `postgresql://...`. The app needs different drivers for different contexts:

| Context | Required Prefix | Where Normalized |
|---------|----------------|-----------------|
| Async app (FastAPI) | `postgresql+asyncpg://` | `backend/app/config.py` (field_validator) |
| Sync migrations (Alembic) | `postgresql+psycopg://` | `backend/alembic/env.py` (normalize_database_url) |

Both normalizers use the same regex pattern to handle any input variant (`postgres://`, `postgresql://`, `postgresql+asyncpg://`, `postgresql+psycopg://`).

### Cron Services

All cron services only need: `DATABASE_URL` = `${{Postgres.DATABASE_URL}}`

**Shared secrets:** Use Railway reference variables (`${{service.VAR}}`) instead of copy-pasting values. One source of truth, auto-synced.

## Cron Jobs

| Name | Schedule | Command | Purpose |
|------|----------|---------|---------|
| `daily-mlb-ingest` | `0 8 * * *` (8am UTC) | `python -m app.scripts.daily_ingest` | Fetch boxscores + seed daily stats |
| `news-scrape` | `0 */3 * * *` | `python -m app.scripts.fetch_player_news` | Player news/injury updates |
| `il-status` | `0 9 * * *` (9am UTC) | `python -m app.scripts.fetch_il_status` | IL status updates |
| `weekly-roster-sync` | `0 21 * * 1` (Mon 9pm UTC) | `python -m app.scripts.weekly_sync` | Refresh players, rosters, draft |
| `draft-monitor` | `*/5 * * * *` | `python -m app.scripts.draft_monitor` | Detect + scrape active drafts |

Cron services spin up, execute, and shut down. Billed only for seconds of execution (~$0.50/mo total).

## Bootstrap (One-Time)

After initial deploy, seed the database:

```bash
railway login
railway link  # select Scoresheet Manager project
railway run -s scoresheet-backend -- python -m app.scripts.seed_all
```

This runs 6 steps: league, teams, users, player list, rosters, draft schedule.

## Deploys

Automatic on push to `main`:
1. Railway detects push, rebuilds affected services
2. Backend: installs deps from `requirements.txt`, runs `alembic upgrade head`, starts uvicorn
3. Frontend: installs deps, builds Next.js, starts `next start` on `$PORT`
4. Health checks pass, traffic swaps (zero-downtime)
5. If health check fails, old instance keeps running

## Troubleshooting

**Pre-deploy fails "command not found":** Check that `requirements.txt` exists and lists the dependency. Railpack won't install packages from `pyproject.toml` alone.

**"No module named X":** Same root cause — missing from `requirements.txt`.

**Frontend can't reach backend:** Verify `BACKEND_URL` uses `*.railway.internal` hostname. Both services must be in the same Railway project.

**OAuth redirect fails:** `NEXTAUTH_URL` must exactly match the Railway public domain (including `https://`). Same URL must be in Google Cloud Console authorized redirect URIs.

**Secrets mismatch:** Use `${{scoresheet-backend.VAR}}` references from the frontend instead of copy-pasting values.

**`railway.toml` vs Dashboard:** The toml file takes precedence. If you override a setting in the Dashboard, it won't take effect if the toml also defines it.
